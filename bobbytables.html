▾ How this payload works

        

          
    
            
  • ' OR '1'='1 closes the string literal early, then adds a condition that's always true — the WHERE clause matches every row, so the query returns the first user regardless of password.
    •

    •

02  UNION-based data extraction

    
A product search box that queries products(name, price). There's a second, unlinked table admin_credentials(username, password_hash) the UI never exposes — but a UNION SELECT with matching columns can pull it into the results.

▾ How this payload works

        

          
    
            
  • The payload starts with a term that matches nothing, closes the string, then appends a second SELECT joined with UNION.
    • 
            
  • For a UNION to work in real SQL, the second query needs the same number of columns as the first — here that's exactly username, password_hash lined up against name, price.
    • 
            
  • The app blindly renders both result sets in the same table, so credentials from an unrelated, unlinked table end up on the storefront page.
    • 
            
  • In secure mode, the whole input is one bound parameter — there's no way to inject a second statement, because it was never treated as SQL syntax to begin with.

03  The fix: parameterized queries

    
The vulnerability isn't "user input" — it's building SQL by string concatenation. Parameterized queries (a.k.a. prepared statements) send the query structure and the data separately, so the database never interprets input as code.
